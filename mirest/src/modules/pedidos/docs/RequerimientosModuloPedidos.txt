# Plan de desarrollo — MiRest con IA  
## Objetivo general

Implementar nuevas funciones en el sistema **MiRest con IA** para mejorar la gestión de pedidos, facturación, impresión, configuración, onboarding, experiencia PWA móvil y seguridad operativa.

El sistema debe evolucionar de una web responsive a una solución más completa, con:

- Gestión real de pedidos de **salón, delivery y para llevar**.
- Flujo correcto de **boleteo / facturación**.
- Configuración de **impresoras para boletero y factura**.
- PWA instalable en Android/iOS con experiencia móvil diferenciada.
- Onboarding PRE / PRO / POST más claro y guiado.
- Mayor seguridad contra acciones indebidas de usuarios internos.
- Preparación para integración futura con IA y base de datos.

---

# 1. Qué te están pidiendo hacer

## 1.1 Pedidos con formato tipo ticket

Cuando se realice un pago en:

- Delivery
- Para llevar
- Salón / mesas

Debe generarse una vista de pedido en formato tipo **ticket**, mostrando información clara para imprimir.

El ticket debe incluir:

- Código o número de pedido.
- Tipo de pedido: Salón, Delivery o Para llevar.
- Fecha y hora.
- Cliente, si aplica.
- Mesa, si aplica.
- Mesero o responsable.
- Productos.
- Cantidad.
- Precio unitario.
- Subtotal.
- Total.
- Método de pago.
- Estado del pedido.
- Estado de facturación.
- Estado de impresión.

También debe mostrarse el boletero asignado:

```txt
Boletero: Epson TM-T20
Estado: Conectado
````

O en caso contrario:

```txt
Boletero: Epson TM-T20
Estado: Desconectado
```

Dentro del dashboard debe existir una sección:

# 4. Onboarding PRE / PRO / POST

## 4.1 PRE Onboarding

El **PRE onboarding** ocurre antes de que el usuario use el sistema por completo.

Debe servir para configurar lo básico:

* Nombre del negocio.
* Tipo de negocio.
* Mostrar bienvenida.
* Mostrar sueldo.
* Frecuencia de sueldo.
* Configuración inicial de impresoras.
* Configuración inicial de mesas.
* Configuración inicial de meseros.

Debe ejecutarse solo una vez, salvo que el usuario decida repetirlo.

---

## 4.2 PRO Onboarding

El **PRO onboarding** es el tutorial guiado del sistema.

Debe poder repetirse desde Configuración.

Debe explicar de forma natural, no técnica, las áreas principales:

* Añadir mesas.
* Asignar meseros.
* Secciones principales.
* Añadir pedidos.
* Paso a paso para añadir pedidos.
* Ver pedidos.
* Estados de mesas.
* Lugar de mesas.
* Concepto de rondas.
* Concepto de pagos.
* Paso a paso de pagos.
* Cómo boletear.
* Cómo facturar.
* Delivery.
* Para llevar.
* Mini secciones del sistema.

Cada paso debe tener:

```txt
Omitir | Anterior | Siguiente
```

También debe existir un botón de ayuda, foco o signo de interrogación para volver a activar guías.

---

## 4.3 POST Onboarding

El **POST onboarding** es la bienvenida cada vez que el usuario entra.

Ejemplo:

```txt
Bienvenido, hoy tienes 8 mesas activas y 3 pedidos pendientes por facturar.
```

Debe ser breve, útil y no invasivo.

* Oscurecer el resto de la pantalla.

Entonces se debe resaltar solo la zona donde aparece el mesero.

# 6. Facturación

## 6.1 Nuevo apartado Factura

Debe existir una sección independiente en el dashboard:

```txt
Dashboard → Factura
```

Este apartado debe tener dos partes:

## Parte superior: formulario de factura

Debe permitir generar una factura con:

* Cliente.
* DNI o RUC.
* Dirección, si aplica.
* Pedido asociado.
* Tipo de comprobante.
* Método de pago.
* Total.
* Observaciones.
* Botón para generar factura.

## Parte inferior: historial de facturas

Debe mostrar las facturas realizadas con datos importantes:

| Código    | Cliente         | Tipo    | Total    | Fecha      | Estado  |
| --------- | --------------- | ------- | -------- | ---------- | ------- |
| F001-0001 | Juan Pérez      | Factura | S/ 85.00 | 25/04/2026 | Emitida |
| B001-0007 | Cliente general | Boleta  | S/ 24.00 | 25/04/2026 | Emitida |

---

## 6.2 Regla importante

Cuando un pedido de cualquier tipo solicite factura, el sistema debe llevar automáticamente al apartado **Factura**.

Flujo:

```txt
Pedido → Solicita factura → Ir a Factura → Completar formulario → Emitir comprobante
```

# 7. Flujo de pedido, cocina, boletero y factura

El flujo principal debe ser:

```txt
Pedido → Cocina → Imprimir en boletero → Pago → Boleta o Factura
```

## Reglas

* Un pedido realizado no debe poder eliminarse si no ha sido facturado o anulado correctamente.
* Si el pedido fue enviado a cocina, debe quedar trazabilidad.
* Si el pedido fue pagado, debe quedar registro.
* Si el pedido fue facturado, debe quedar bloqueado para edición común.
* Solo roles autorizados pueden anular, corregir o eliminar.

## 11.2 Diferencia entre responsive y PWA mobile

No basta con que la web se vea pequeña en celular.

Se pide una versión mobile con UI/UX diferente.

Esto significa:

| Web responsive                | PWA mobile                            |
| ----------------------------- | ------------------------------------- |
| Misma web adaptada al tamaño  | Experiencia diseñada para celular     |
| Navegación parecida a desktop | Navegación mobile tipo app            |
| Menús reducidos               | Bottom navigation o navegación mobile |
| Paneles adaptados             | Pantallas específicas                 |
| Vista compacta                | Flujo más directo y táctil            |

---

## 11.3 Comportamiento esperado

Cuando se abra en desktop:

```txt
Mostrar dashboard web normal.
```

Cuando se abra desde mobile:

```txt
Mostrar experiencia PWA mobile distinta.
```

La lógica funcional debe mantenerse, pero la interfaz debe cambiar.

# 15. Seguridad y roles

## 15.1 Problema a resolver

Se debe evitar que un mozo o usuario interno pueda manipular pedidos, eliminar ventas o alterar información sin autorización.

## 15.2 Reglas de seguridad

Implementar roles como:

| Rol                 | Permisos                                 |
| ------------------- | ---------------------------------------- |
| Admin               | Control total                            |
| Cajero / Facturador | Cobrar, facturar, imprimir               |
| Mesero              | Crear pedidos, ver mesas asignadas       |
| Cocina              | Ver pedidos enviados a cocina            |
| Supervisor          | Revisar historial, autorizar anulaciones |

---

## 15.3 Acciones protegidas

Deben requerir permiso especial:

* Eliminar pedido.
* Anular pedido.
* Editar pedido enviado a cocina.
* Cambiar total.
* Reimprimir comprobante.
* Cambiar método de pago.
* Modificar factura.
* Desconectar impresora.
* Cambiar configuración crítica.

---

## 15.4 Trazabilidad

Cada acción importante debe guardar:

* Usuario que realizó la acción.
* Fecha y hora.
* Módulo.
* Acción realizada.
* Pedido afectado.
* Motivo, si aplica.

Ejemplo:

```txt
Usuario: Pedro Mesero
Acción: Intentó eliminar pedido enviado a cocina
Pedido: M-04-R2
Fecha: 25/04/2026 19:45
Resultado: Bloqueado por permisos
```

# 18. Capa gratuita y estrategia comercial

Como product designer, se debe establecer una capa gratuita para atraer clientes.

Puede incluir:

* Gestión básica de mesas.
* Pedidos limitados.

## 18.2 Escalera de valor tipo Russell Brunson

La idea es vender por niveles:

Lead Magnet → Oferta económica → Plan básico → Plan Pro → Plan premium

Ejemplo:

| Nivel       | Producto                           |
| ----------- | ---------------------------------- |
| Lead Magnet | Diagnóstico gratis del restaurante |
| Bajo costo  | Menú digital simple                |
| Básico      | Sistema de pedidos                 |
| Pro         | Facturación + delivery + reportes  |
| Premium     | IA + predicción + automatización   |

# 22. Resumen técnico para pasar a programación

Se debe implementar una evolución completa del dashboard actual, priorizando primero la arquitectura del sistema y luego la experiencia visual.

La implementación debe dividirse en módulos:

```txt
/configuracion
/factura
/impresoras
/onboarding
/pedidos
/mesas
/delivery
/para-llevar
/pwa
/seguridad
/auditoria
```

Cada módulo debe tener lógica separada, estado claro y preparación para base de datos.

El enfoque principal no es solo agregar pantallas, sino crear un flujo sólido:

```txt
Pedido → Cocina → Boletero → Pago → Boleta/Factura → Historial → Auditoría
```

La prioridad debe ser que el sistema sea:

* Rápido.
* Claro.
* Seguro.
* Escalable.
* Fácil de usar por meseros, cajeros y administradores.
* Preparado para funcionar como PWA instalable.
